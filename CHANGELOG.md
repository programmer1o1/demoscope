# Changelog

## v0.5.0

The biggest release yet: demoscope stops being Source-only and grows two entire new engine families, finishes the proto-4 Source work, and renders maps for all of them.

- **Two new engine families - Quake and GoldSrc.** The viewer now decodes the **Quake family** (Quake 1 NetQuake, Quake 2, Quake 3 Arena) and **GoldSrc** (Half-Life 1 / CS 1.6 / DoD / Condition Zero) through the exact same 3D viewer, minimap, heatmap, and POV camera as Source. Quake demos are server-stream recordings (positions are already in the bytes); GoldSrc's `HLDEMO` container was reverse-engineered from scratch and verified byte-for-byte, with the recorder POV camera pulled from each NetMsg `RefParams` - **no IDA needed, GoldSrc's structs are public** (transcribed from the open-source `hldemo` parser).
- **Maps for everyone - Quake 1/2/3 + GoldSrc BSP overlays.** `bsp::extract_any_bsp` dispatches a `.bsp` by magic/version and renders the world geometry behind the player paths: GoldSrc/Q1 (version 30/29), Quake 2 (`IBSP` v38), Quake 3 (`IBSP` v46, meshvert-indexed). Verified against real maps (`cs_assault`, `dm6`, `base64`, `cpm4`); on the cpm4 demo the recorder track sits 100 % inside the decoded map. Drop the `.bsp` beside the demo, same as Source.
- **Proto-4 Source finished.** Left 4 Dead 1 & 2 now decode **per-player entity tracks** (the last missing piece was L4D's net-message renumbering, read out of `engine.dll` over the IDA Pro MCP bridge); Portal 2 mid-game / SAR speedrun demos decode (the 12-bit `svc_UserMessage` width); **proto-4 game events** extract on Portal 2 / Stanley / L4D; and Garry's Mod (`GMODEMO`) demos are accepted as Source.
- **Kill arcs.** A new **Kills** toggle draws attacker→victim bezier arcs at each kill during playback - headshots in gold, body shots orange-red - with an on-screen color legend.
- **Viewer + codebase.** Single-POV and multi-player rendering were unified behind one path (fixing GoldSrc/GMod/single-player avatar and route bugs, incl. demos where the player starts alive); and the 2,979-line `main.rs` was split into nine cohesive modules (`constants`, `bytes`, `bitreader`, `header`, `events`, `bsp`, `packets`, `json`, `html`).

See the per-item breakdown under **Completed work → v0.5.0** below, and [docs/COMPATIBILITY.md](docs/COMPATIBILITY.md) for the full support matrix.

## v0.4.0

- **Left 4 Dead 1 & 2 now parse end-to-end** - the full demo stream walks to `DEM_STOP` (inputs, names, recorder camera path; `data23.dem` = 509 usercmds, `ellisfloor.dem` = 432). Two container quirks were peeled back: L4D ships **4 splitscreen slots** (`Split_t[4]`, 304-byte `democmdinfo`), and its newer command enum inserts `dem_customdata` at 8 and pushes `dem_stringtables` to **9** (Portal 2 / Orange Box keep it at 8). The **SendTable/DataTable format was also cracked** by sweeping flag-width × priority × `m_nBits` for a sane server-class count: L4D1 uses 16-bit TF2 flags with **`m_nBits = 6`** (222 classes); L4D2 uses 19-bit Alien-Swarm flags + priority with `m_nBits = 6` (278 classes). The shared L4D quirk is the 6-bit `m_nBits` field - everything else uses 7, and a one-bit miscount desyncs the whole table walk. The previously-conflated `portal2_engine` boolean is now decoupled into three independent axes (`portal2_extra_bits` flag format, `bit_count_bits`, and container `is_portal2_engine`). Per-player entity *tracks* still need the L4D net-message map (delivered in v0.5.0).
- **CS:GO parses end-to-end** - the command-enum shift (stringtables at 9) and the splitscreen handling apply, so CS:GO walks to STOP with full inputs and player names. Its `player_info_s` blob has a distinct layout (16-byte `version + xuid` prefix, **128-byte name field, big-endian integers**) - the userinfo parser now detects it by blob size and reads names/SteamIDs correctly. Entity tracks need a protobuf decoder (its DataTables and PacketEntities are protobuf-encoded) - a separate, larger effort.
- **First-person POV camera now uses the real per-frame camera** - on demos with no usercmds (Portal 2, Stanley), the view angle is sourced from the dense `democmdinfo.viewAngles` stream (≈1 sample per game packet) instead of the sparse networked eye-angle SendProp (which a recorder barely networks for itself). Interpolating the dense stream gives a smooth, accurate POV that turns at the camera's real cadence - no slow "smoothing-cam" sweep, no freeze-then-snap. Exposed as `__VIEW_ANGLES__`/`viewAngleAt()`.
- **Smoother camera & timeline** - OrbitControls inertia is off and the follow-target snaps (1:1 tracking, no glide); the timeline playhead is **tick-based** and redraws every frame so it advances continuously even on sparse synthesized timelines (it used to freeze between samples); and **dragging the timeline seeks to a continuous tick** with the avatar/camera interpolated to it, so scrubbing is smooth instead of snapping to sparse cmd samples.
- **Player-name parsing across engines** - `parse_player_info_blob` now handles all three `player_info_s` layouts by blob size: proto-3 (`name[32]`@0), proto-4 (xuid + `name[32]`@8), and CS:GO (version + xuid + `name[128]`@16, big-endian). The dead-player camera-follow case is fixed too (it no longer chases a dead primary's spectated coordinates).
- **Overflow-hardened demo walkers** - every length read in the packet walkers (`main.rs` + `player_tracks.rs`) now guards a negative/garbage length and uses saturating arithmetic. This fixes debug-build panics (`attempt to add with overflow`) on desync; release builds silently wrapped before. Validated across a **450-demo corpus** (TF2, HL2 + episodes, DoD:S, CS:S, CS:GO, Portal 1/2, Stanley, L4D1/2, plus Source 2 cs2/dota2) with **zero crashes** - the 10 non-parsing files are correctly rejected Source 2 (`PBDEMS2`) and HL2-beta (`HLDEMO`) magic.

## v0.3.0

- **Portal 2 & The Stanley Parable now decode real entity positions** - proto-4 (`demo_protocol = 4`) entity decode works end-to-end. `youareamoron.dem` yields 242 position samples of real `sp_a2_core` coordinates; Stanley Parable yields 2226. The last bug was a priority-sort mismatch in the SendTable flatten, found by hooking `engine.dll`'s `CDeltaBitsReader::ReadNextPropIndex` with Frida and diffing the engine's per-prop bit widths against ours. Full account in [docs/PROTO4.md](docs/PROTO4.md).
- **Entity-only playback timeline** - demos with no parsed usercmds (all of proto-4) now synthesize a playback timeline + world positions from the decoded primary-entity track, so the 3D scrubber, follow-cam, and speedometer run on real data instead of throwing on an empty `CMDS` array. The first-person camera pitch comes from `m_angEyeAngles[0]` (yaw from `[1]`), so look-up/down works.
- **Proto-4 player names** - `player_info_s` on Portal 2 / Stanley prepends an 8-byte `xuid` before `name[32]`, so the TF2-layout parser read empty/garbage names. The blob parser now auto-detects the name offset (0 for proto-3, 8 for proto-4), so Portal 2 / Stanley players show their real names (`Dragonツ`, `that mf`).
- **In-browser parser (WASM)** - `src/lib.rs` exposes `parse_demo_to_html` via `wasm-bindgen`; `scripts/build-wasm.sh` produces a 600 KB `.wasm` + JS glue under `web/`. The drag-and-drop UI at `web/index.html` lets users drop a `.dem` (and optionally `.bsp`) onto the page and view it without running the CLI.
- **CS:S parity work** - three concrete fixes for the Source 1 sister game:
  - **Velocity-based teleport detection** - the per-player path-line and `mpSampleAt` interpolation use `(distance / Δt) > 900 u/s` instead of a flat distance threshold. CS:S samples are 1-4 seconds apart so a flat threshold flagged normal running as teleports. Cuts false teleports on `111.dem` from 1.3 % → 0.4 % of sample-pairs.
  - **Sub-second `m_lifeState` flicker filter** - CS:S pulses `m_lifeState=1` for ~20-30 ticks on hit-flinch / weapon-drop events that aren't real deaths. Any dead gap shorter than 0.6 s in the alive-interval list is coalesced into the surrounding alive interval. Real deaths (≥ 1 s dead) survive.
  - **Spec/observer hide** - new `Hide specs ✓` toggle in the Players panel suppresses entities with < 3 position samples that aren't the current primary. Catches idle bots, the HLTV/SourceTV camera, and the recorder when they're spectating someone else (cross-referenced against the `svc_SetView` switch intervals). On by default; right-clicking a sparse-sample entity to set them as primary auto-reveals them.
- **Fire-tracer bug fix** - `mpPrimaryPositionAt` was using the live `currentCmdIdx` for sparse-sample primaries instead of the cmd-index of the requested historical tick. All fire markers in CS:S piled up at the recorder's tick-0 spawn position. Fixed by routing through `findCmdIdx(tick)`.
- **Multi-player by default** - every `--html` run decodes the position, name, and life state of every player entity in the demo. No flag, no feature, no external dependency: the native `source_demo` decoder is the only one shipped.
- **Mid-demo renames + disconnect-reconnect** - `svc_UpdateStringTable` is now decoded for the `userinfo` table, so a player who renamed `name "Lunascape" → "Sierra"` mid-match shows up under their final name. Prior aliases are preserved per slot and displayed as `was X, Y` next to the current name. Primary-entity detection matches the recorder's header nick against any alias.
- **Spectator avatar tracks the camera** - the spec recorder's `m_vecOrigin` only updates at spawn, so demoscope falls back to `democmdinfo.viewOrigin` for sparse-sample entities. Their 3D avatar, route line, minimap dot, and Follow-camera target all use the camera trajectory.
- **BSP displacement support** - terrain surfaces (`LUMP_DISPINFO` + `LUMP_DISP_VERTS`) are now tessellated into the rendered mesh using qbyte's SourceImporter algorithm. Maps like `d2_coast_07` go from boxy brushwork to full hilly terrain.
- **Console panel** - replaces the old Chat panel. Combines chat + kills + spawns + round transitions + connect/disconnect / name-change lines into a single Source-style log. Click any line to seek to that tick; category chips filter what's shown; the `⛶` button pops it out into a fullscreen overlay that auto-scrolls during playback.
- **GIF export** - `Record GIF` button captures the live 3D scene as an animated GIF, defaulting to first-person camera at 10× playback for ~10s of compressed action.
- **Real player positions** - extracted from `democmdinfo.viewOrigin` (single-POV) and per-entity `m_vecOrigin` SendProps (multi-player), so rocket jumps, teleporters, and all physics are captured exactly.
- **Full 3D BSP map overlay** - entire map mesh rendered alongside player paths; supports LZMA-compressed lumps (common in TF2 workshop maps).

## Completed work (by difficulty tier)

### v0.5.0

- ✅ **Quake 1 / 2 / 3 BSP map overlays** - the 3D viewer now renders the actual map for Quake demos when the matching `.bsp` is beside the demo (`bsp::extract_any_bsp` dispatches by magic/version). **Quake 1** (version 29) reuses the GoldSrc lump walk; **Quake 2** (`IBSP` v38) is the same edge/surfedge fan-triangulation with a 19-lump header, 76-byte texinfo and SURF-flag sky/nodraw culling; **Quake 3** (`IBSP` v46) is a different model - 44-byte drawverts indexed by per-face meshverts (polygon + mesh faces), with bezier patches counted-and-skipped and tool/sky shaders dropped by name. Verified directly (`dm6.bsp` 6750 tris, `base64.bsp` 64962 tris, `cpm4.bsp` 10711 tris - Q2/Q3 match an independent Python reference exactly) and end-to-end on the cpm4 demo, where the recorder track sits **100% inside** the decoded map bounds. A `#[cfg(test)]` regression test decodes all three. Same "drop the map next to the demo" model as Source/GoldSrc.
- ✅ **Kill arcs (attacker → victim)** - a new **Kills** toggle draws a bezier arc from the attacker's position to the victim's at the moment of each kill, lit up during playback (a quick snap-in then a `KILL_FADE_SEC` fade-out, the same age-window idea the fire tracers use), with **headshots in gold** and body shots in orange-red plus an impact dot at the victim. Both endpoints are sampled from the per-player entity tracks at the death tick via the userid→entity map, so it needs MP entity data (no-op on single-POV demos). Verified on `111.dem` (CS:S, 26 players): 300 `player_death` events → **281 arcs** (82 headshots), 2 suicides and 17 disconnected-player deaths correctly skipped, arc lengths 55-2984 units. `player_hurt` shares the same path (faint, non-fatal) where a demo carries it.
- ✅ **GoldSrc (`HLDEMO`) support - container + recorder POV camera** - Half-Life 1 / CS 1.6 / DoD / Condition Zero demos decode in `src/goldsrc.rs`. The container was reverse-engineered and verified byte-for-byte against a real Condition Zero demo (`cs_assault`, `czero`, demo proto 5 / net proto 48, **95.78 s, 7916 frames**, directory math exact): the 544-byte header (map, game dir, protocols), the directory offset at byte 540, the 92-byte directory entries, and the per-segment frame stream (9-byte `type/time/frame` header; types 0/1 NetMsg, 2 DemoStart, 3 ConsoleCommand, 4 ClientData, 5 NextSection, 6 Event, 7 WeaponAnim, 8 Sound, 9 DemoBuffer). The **recorder camera path now renders** - eye origin + view angles are pulled from each NetMsg `RefParams` (`vieworg` at body+4, `viewangles` at body+16, `msg_length` at body+464). The frame layout and struct field order were transcribed from the open-source `hldemo` parser (no IDA - GoldSrc's structs are public, unlike Source's closed `engine.dll`). A handful of frames per demo carry a length quirk that desyncs the cursor; the walker resyncs to the next `RefParams`-validated NetMsg (≈26 / 24k frames on the sample), so the trajectory stays complete: `cs_assault` yields **7916 camera samples** spanning the full map (X[-1009,807] Y[156,2996] Z[30,392]) with realistic pitch [-20°,27°]. The recorder is fed as a real entity track (id 1), not the usercmd-indexed single-POV path - GoldSrc has no usercmds, so that path's `finalPositions` would be empty. The **GoldSrc BSP map overlay** also renders (`extract_goldsrc_bsp_from_bytes`, version 30 / Quake-1 version 29): the same lump-based face walk as VBSP with a 15-lump header, 20-byte `dface_t`, 40-byte texinfo and no LZMA/displacements; sky + tool brushes dropped by texture name. Verified on `cs_assault.bsp` - 2805 verts / 5547 tris, geometry enveloping the recorder path exactly. Drop the `.bsp` next to the demo like Source. Per-player entity tracks (the *other* players) remain on the roadmap. See [docs/COMPATIBILITY.md](docs/COMPATIBILITY.md).
- ✅ **`main.rs` split into cohesive modules** - the 2979-line monolith was carved into `constants`, `bytes`, `bitreader`, `header`, `events`, `bsp`, `packets`, `json`, and `html` (main.rs is now a 453-line CLI shell). Pure mechanical refactor - function bodies moved verbatim, all three build configs (bin / native lib / wasm32) compile, and the full demo corpus produces byte-identical output with 0 panics. New modules are declared inside `main.rs` so they inherit its dual-build wiring (`#[path]`-included as `mod cli` by `lib.rs`); cross-module paths use portable `super::`/`self::`.
- ✅ **Garry's Mod 13+ (`GMODEMO`) accepted as a Source demo** - GMod's demo container is byte-for-byte identical to HL2DEMO (same 1072-byte header, same proto-3 packet stream); only the 8-byte magic differs (`GMODEMO\0` vs `HL2DEMO\0`). It was being rejected on the magic check and, because of its `.dem` extension, misrouted to the NetQuake decoder. Now the magic is accepted everywhere (`is_source_demo_magic`) and excluded from the Quake gate, so GMod demos decode their **inputs, game events, names, and recorder camera path**. Entity tracks remain 0 (GMod reports `demo_protocol=3` but networks entities in the newer proto-4 style, so the proto-3 entity decoder desyncs - DataTables parse at 248 classes and `svc_PacketEntities` is reached, but no entities decode).
- ✅ **Unified the viewer's single-POV and multi-player rendering** - demos with no networked entity tracks (single-player HL2/Portal, GMod sandbox) used to fall back to a separate legacy recorder avatar; now the recorder is synthesized as one normal sparse entity driven by the camera trajectory, so every demo renders through the single multi-player path. Fixed the GMod avatar vanishing mid-playback: a sole/sparse recorder is flagged `soloRecorder` and skips the spec / `svc_SetView` / life-state hiding that only makes sense when there are *other* players to avoid overlapping (demos with real per-player tracks - TF2/CS:S/L4D/Portal 2 - are untouched, so that hiding still applies there). The same `soloRecorder` rule keeps the sole avatar visible through death on single-player demos (demos with exactly one player entity), while genuinely multiplayer demos - verified 26-player CS:S and 4-player L4D2 - still hide dead/spectating players.
- ✅ **Fixed blank routes / frozen follow-camera / hidden avatar on demos where the player starts alive** - the alive-interval builder only *opened* a life interval on a "becoming alive" (`m_lifeState == 0`) transition. But `m_lifeState` is networked only when it *changes*, so a recorder that is alive from the start (every single-player HL2/Portal demo, and any player already in the server when recording began) records its **first** transition as the death itself, with no preceding alive marker - yielding **zero** alive intervals. That one bug simultaneously blanked the entity's route (`mpShownIvIndexAt` returned -1 for every sample, so no path runs were built), froze the follow camera (`mpPrimaryDeadAt` read the player as perpetually dead), and hid the avatar. A leading death-transition is now treated as "alive since tick 0." Multiplayer life counts are unchanged (a leading *spawn* transition still behaves as before) - verified on 26-player CS:S.
- ✅ **Proto-4 game-event extraction (Portal 2 / Stanley / L4D)** - made the demo-command and net-message walkers proto-4-aware so game events are decoded wherever the stream actually carries them. Four bugs, all now aligned with `player_tracks::scan_game_payload`:
  - **`DEM_CUSTOMDATA` misframe** - proto-4 inserted cmd 8 (`id(4)+length(4)` header); three walkers (`iterate_demo_packets`, `parse_userinfo_from_demo`, the CLI display loop) read its `id` as a bare `length` and desynced the demo-command stream right after the signon. On Portal 2 this dropped **every** game packet (game-packet collection 0 → 2231 on `testingportal`), which also broke its `--csv`/`--json`/`--all` CLI output; now the walk reaches `DEM_STOP`.
  - **Splitscreen mis-probe** - `detect_splitscreen` probed `[4,2,1]` and could false-positive as 4 on Portal 2 (real value 2); now pinned to 2 for Portal 2-engine games, matching `player_tracks`.
  - **No net-message remap** - the event parser (`extract_events_from_payload`) was proto-3-only, so the per-packet `net_Tick` (raw 4) was misread; now it applies the same `remap_msgs` / `user_msg_12bit` logic.
  - **`break` on `svc_Sounds`** - the event parser bailed on `svc_Sounds` (17), `svc_UpdateStringTable` (13), and the Portal 2 `svc_CmdKeyValues`/`svc_PaintMapData` (32/33), which lead most proto-4 game packets - so it never reached `svc_GameEvent` (25). Those are now framed properly; the walker stays synced through `svc_PacketEntities` (verified 2136× in sync on Portal 2).
  - **Result:** L4D2 `demogaty.dem` displayed events 322 → 324 (the walker now reaches all 52 networked `svc_GameEvent` messages cleanly; most are event types the default per-game display filter intentionally drops). Proto-3 (CS:S `111.dem`) unchanged at 3175 (a few more recovered by the corrected `svc_Sounds` framing - no regression). Single-player Portal 2 / Stanley demos report 0 game events, which is **correct**: they carry no networked `svc_GameEvent` messages at all (verified - zero id-25 in the stream; single-player events fire locally and aren't serialized).
- ✅ **L4D1 / L4D2 per-player entity tracks** - the last missing layer was the **net-message map**. L4D uses the same renumbering as the Portal 2 engine (`NetSplitScreenUser` at 3 ⇒ `NetTick = 4`, `SvcPrint = 16`, `SvcSplitScreen = 22`), so the per-packet `net_Tick` was being misread as `net_StringCmd` and the cursor desynced before any `svc_PacketEntities` - hence the long-standing "0 tracks". Verified by reading L4D1's `engine.dll` over the IDA Pro MCP bridge: `NET_Tick::GetType()` returns 4, `SVC_Print` 16, `SVC_UserMessage` 23. The remap is now applied to L4D as well as Portal 2 - but kept **independent** of the 12-bit `svc_UserMessage` width, because L4D's `SVC_UserMessage::ReadFromBuffer` reads `v & 0x7FF` (**11 bits**, not Portal 2's 12). `demogaty.dem` (L4D2, `c8m2_subway`) now decodes **2 entity tracks / 447 position samples / 4 named players / weapon switches**, 99 % clean (1322 ok / 12 benign). Portal 2 and proto-3 (TF2/CS:S) unchanged.
- ✅ **Portal 2 mid-game / speedrun demos decode** - `svc_UserMessage`'s length field is **12 bits on the Portal 2 engine, not 11** (the older-Source width the parser used). One bit off desyncs the message cursor the instant a user message appears - and a SAR fullgame run (`fullgame_…finale4.dem`) carries one in the **tick-0 signon packet**, right before its only full entity baseline (recording started mid-game), so nothing decoded → **0 samples, blank viewer**. The same bug had been silently truncating `testingportal` after ~750 ticks. Fixed in `scan_game_payload` and gated on the Portal 2 engine flag (`len_bits = if portal2 {12} else {11}`), so proto-3 TF2/CS:S are byte-identical to before. `fullgame` 0 → 2485 samples; `testingportal` 1773 → 1802. Verified by reading `SVC_UserMessage::ReadFromBuffer` straight out of `engine.dll` over the **IDA Pro MCP bridge** (`m_nLength = v & 0xFFF`) - and while there, cross-checked the entity-header reader and the deletions loop against our decoder (both already correct).
- ✅ **Quake family support (`src/quake/`)** - demoscope is no longer Source-only. **Quake 2** (`.dm2`, protocol 34) and **Quake 3 Arena** (`.dm_68`, protocol 68) decode real per-player position tracks, viewangles, names, and map, rendering through the existing 3D viewer; **Quake 1** NetQuake (`.dem`, protocol 15) is implemented to spec. Q3's Huffman tree + field tables are ported verbatim from GPL ioquake3 with the frequency table byte-checked. See [docs/COMPATIBILITY.md](docs/COMPATIBILITY.md#quake-family).
- ✅ **L4D2 BSP overlay** - the v21 lump directory is auto-detected (some v21 maps shift it from offset 8 to 12); `detect_lump_base` scores both and prefers the standard 8, so v20 maps don't regress. Fixed a WASM panic (`negate overflow` on an `i32::MIN` surfedge + an empty displacement-corner index).
